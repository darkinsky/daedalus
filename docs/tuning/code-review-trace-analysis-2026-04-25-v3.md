## 3. 新问题：上下文 150K 峰值后持续下降到 80K

### 根因

v2 引入的 `TruncationConfig` 使用**固定分层策略**（基于轮次距离）：
- 最近 10 轮完整 → 倒数 11-20 轮截断到 4000 字符 → 更老的截断到 1500 字符

问题：当历史超过 13 轮时，第 1-3 轮自动进入 "1500 字符" aggressive 区，
即使此时总上下文才 153K（远低于 200K 目标预算）。这是"距离截断"的固有缺陷：
它**不关心是否真的需要截断**，只看轮次距离。

### 修复

将截断算法改为**基于 token 预算的动态策略**：

1. 估算全部历史的 token 总量
2. 如果低于预算（200K × 60% = 120K tokens）→ **不截断，全部保留**
3. 如果超出预算 → 从最老的轮次开始，逐级应用 moderate → aggressive → micro 截断
4. 每级截断后重新检查是否回到预算内，是则停止

关键改动：最近 `min_recent_rounds=10` 轮永远不截断。

### 预期效果

上下文将持续增长直到接近 120K token 预算（~480K 字符），
然后通过渐进截断稳定在 **100-120K tokens**，而非 80K。
